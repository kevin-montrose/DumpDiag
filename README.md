# DumpDiag

Pre-Release - this probably has loads of bugs!

**Note: currently only supports x64 processes**

## Install

`dotnet tool install DumpDiag`

DumpDiag assumes [`dotnet-dump`](https://docs.microsoft.com/en-us/dotnet/core/diagnostics/dotnet-dump) or WinDbg (local or remote) are also installed.

## Options

Run with `dumpdiag` after installation, assuming global installation.

To use DumpDiag with `dotnet-dump`, run with the `dotnet-dump` option.

```
Usage:
  DumpDiag [options] dotnet-dump

Options:
  -ddp, --dotnet-dump-path <dotnet-dump-path>        Path to dotnet-dump executable, will be inferred if omitted
                                                     [default: C:\Users\kevin\.dotnet\tools\dotnet-dump.exe]
  -dfp, --dump-file-path <dump-file-path>            Existing full process dump to analyze [default: ]
  -dpid, --dump-process-id <dump-process-id>         Id of .NET process to analyze [default: ]
  -dp, --degree-parallelism <degree-parallelism>     How many processes to use to analyze the dump [default: 15]
  -sdp, --save-dump-file-path <save-dump-file-path>  Used in conjunction with --dump-process-id, saves a new full
                                                     process dump to the given file [default: ]
  -rfp, --report-file-path <report-file-path>        Instead of writing to standard out, saves diagnostic report to the 
                                                     given file [default: ]
  -mc, --min-count <min-count>                       Minimum count of strings, char[], type instances, etc. to include
                                                     in analysis [default: 1]
  -mas, --min-async-size <min-async-size>            Minimum size (in bytes) of async state machines to include a field 
                                                     breakdown in analysis [default: 1]
  -o, --overwrite                                    Overwrite --report-file-path and --dump-file-path if they exist
                                                     [default: False]
  -q, --quiet                                        Suppress progress updates [default: False]
  -?, -h, --help                               Show help and usage information
```

To use WinDbg, run with the `windbg` option.  Note that WinDbg must be [running as a server](https://docs.microsoft.com/en-us/windows-hardware/drivers/debugger/-server--create-debugging-server-), accepting remote connections over TCP.

```
Usage:
  DumpDiag [options] windbg

Options:
  -wcs, --windbg-connection-string             Connection details for remote WinDbg session, formatted like <ip>:port
  <windbg-connection-string>
  -dedp, --dbgeng-dll-path <dbgeng-dll-path>   Path to dbgeng.dll matching remote WinDbg version, will be inferred if
                                               omitted [default: C:\Program Files (x86)\Windows
                                               Kits\10\Debuggers\x64\dbgeng.dll]
  -rfp, --report-file-path <report-file-path>  Instead of writing to standard out, saves diagnostic report to the given 
                                               file [default: ]
  -mc, --min-count <min-count>                 Minimum count of strings, char[], type instances, etc. to include in
                                               analysis [default: 1]
  -mas, --min-async-size <min-async-size>      Minimum size (in bytes) of async state machines to include a field
                                               breakdown in analysis [default: 1]
  -o, --overwrite                              Overwrite --report-file-path if it exists [default: False]
  -q, --quiet                                  Suppress progress updates [default: False]
  -?, -h, --help                               Show help and usage information
```

Note that you can specify multiple `--windbg-connection-string` arguments to split analysis between multiple remote sessions.  This allows similar scaling to `dotnet-dump`'s `--degree-parallelism` option.

## Remarks

DumpDiag performs routine analysis of a dump of a .NET process using `dotnet-dump`:

 - Counts of values on the heap, by type
 - Counts of strings on the heap, by content
 - Counts of char[]s on the heap, by content
 - Counts deleges on the heap, by backing method
 - Counts of async state machines, by "backing" method
 - Locals captured by async state machines
 - All managed thread call stacks
 - Counts of unique stack frames from managed threads
 - Heap fragmentation by generation, LOH, and POH
 - Pinned objects including by generation, LOH, and POH

Where relevant total, live, and dead counts are reported.  High % of dead references (especially over time) can indicate inefficient allocation patterns.

High fragementation in a generation, coupled with large numbers of pinned objects in that generation, can indicate holding pins for too long.

Constraints for `dotnet-dump` apply equally to DumpDiag, primarily that dumps must be analyzed on the same OS as they were captured on.
